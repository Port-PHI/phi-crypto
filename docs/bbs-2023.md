# The `bbs-2023` interoperability suite

`bbs_2023` implements the W3C [`bbs-2023`](https://www.w3.org/TR/vc-di-bbs/) cryptosuite, which
builds on the IRTF CFRG [BBS signature scheme](https://datatracker.ietf.org/doc/draft-irtf-cfrg-bbs-signatures/).
It is the standards-interoperable companion to the project's `phi-bbs-v1` suite.

- **Ciphersuite:** `BBS_BLS12381G1_XMD:SHA-256_SSWU_RO_H2G_HM2S_` (BLS12-381, SHA-256).
- **Backend:** the audited `pairing_crypto` crate (MATTR/DIF), pinned to `v0.4.4`. We do not
  re-implement the ciphersuite.
- **Conformance:** verified byte-for-byte against the official IETF/W3C test vectors — see
  `tests/bbs_2023_kat.rs` and `tests/vectors/bbs_2023_sha256/` (provenance in that folder's
  `SOURCE.md`).

## Why native only

`pairing_crypto` uses the `blst` C backend, which does not cross-compile to
`wasm32-unknown-unknown` without a wasm-capable C toolchain. To keep the WASM build simple and
pure-Rust, `bbs_2023` is gated to non-`wasm32` targets; the browser build keeps `phi-bbs-v1`
(which is pure Rust and compiles to wasm with no extra toolchain).

## API

Two layers are exposed:

1. **Raw IRTF operations** with an explicit `header` — used for interop and known-answer tests:
   `sign`, `verify`, `proof_gen`, `proof_verify`.
2. **`Bbs2023Sha256`**, an implementation of the project's `CryptoSuite` trait. It maps the uniform
   selective-disclosure API onto this ciphersuite:
   - the signature `header` is empty;
   - the `nonce` argument is the BBS *presentation header*;
   - to keep the trait's `derive_proof` (which is not given the public key) working,
     `sign_credential` returns `public_key ‖ signature`.

## Relationship to `phi-bbs-v1`

`phi-bbs-v1` (module `bbs`) remains the default suite and is available on every target, including
wasm. `bbs-2023` is selected explicitly when standards interoperability is required. Both implement
the same `CryptoSuite` trait.
